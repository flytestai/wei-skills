# 自动化测试题

### 1. 接口自动化测试用例一般怎么设计
- tags: 接口自动化, 用例设计, 高频
- difficulty: easy
- prompt: 如果让你设计一组接口自动化测试用例，你一般会从哪些维度去考虑？
- answer: 我一般会先从功能正确性开始，也就是正常入参、正常业务流程是否通过；然后再补参数校验，比如必填项、字段类型、长度、枚举值、边界值；再看异常场景，比如非法参数、重复请求、权限不足、token 失效、依赖服务异常；如果是核心接口，还会补充幂等性、并发、数据一致性，以及必要的响应时间观察。简单说，我会按“正常 + 异常 + 边界 + 鉴权 + 业务约束 + 稳定性”几个层面来设计。

### 2. 接口自动化里，断言一般断哪些内容
- tags: 接口自动化, 断言设计, 高频
- difficulty: easy
- prompt: 做接口自动化时，你一般会断言哪些内容？
- answer: 我一般不会只断言 status code，而是分层断言。第一层是 HTTP 层，比如状态码、响应时间、header；第二层是业务层，比如 code、message、success 这些业务字段；第三层是数据层，比如返回体字段值是否正确、数据库是否落库、上下游状态是否一致；如果接口有副作用，我还会校验调用前后数据变化。也就是说，接口自动化断言应该覆盖“接口返回对不对”和“业务结果对不对”两层。

### 3. 什么样的接口适合优先做自动化
- tags: 接口自动化, 自动化策略, 高频
- difficulty: easy
- prompt: 在项目里，你会优先把哪些接口纳入自动化测试？
- answer: 我会优先做高频回归、业务核心链路、容易出错、人工执行成本高、且结果相对稳定的接口。比如登录、下单、支付前置、用户信息查询、状态流转这类核心接口。相反，一些需求变化特别快、返回不稳定、或者一次性验证价值不高的接口，我会先放后面。因为自动化不是越多越好，而是要优先覆盖回归价值高的部分。

### 4. UI 自动化为什么容易 flaky
- tags: UI自动化, flaky, 高频
- difficulty: easy
- prompt: UI 自动化测试为什么经常容易出现 flaky case？
- answer: UI 自动化容易 flaky，核心原因是它依赖页面渲染、元素定位、网络状态和执行环境，这些都比接口层更容易波动。常见原因有：元素定位方式不稳定、页面异步加载导致时序问题、固定 sleep 使用过多、测试数据相互污染、浏览器或环境差异、弹窗和动画干扰等。所以 UI 自动化不稳定，本质上很多时候不是断言错了，而是定位和等待策略不够稳。

### 5. UI 自动化里怎么减少 flaky
- tags: UI自动化, 稳定性优化, 高频
- difficulty: medium
- prompt: 如果 UI 自动化经常不稳定，你会从哪些方面去优化？
- answer: 我一般会从四个方面优化。第一是定位策略，优先用稳定的业务标识，不用容易变化的 xpath 层级路径；第二是等待机制，尽量显式等待替代 sleep；第三是数据和环境隔离，避免前后用例互相影响；第四是失败可观测性，比如截图、日志、页面源码、步骤日志，方便快速定位。我的经验是，UI 自动化稳定性更多是工程问题，不只是写脚本的问题。

### 6. UI 自动化和接口自动化怎么分工
- tags: UI自动化, 接口自动化, 测试分层, 高频
- difficulty: medium
- prompt: 在项目里，UI 自动化和接口自动化一般应该怎么分工？
- answer: 我的理解是接口自动化应该承担主回归，因为执行快、稳定性高、维护成本低；UI 自动化更适合覆盖少量关键端到端链路，比如登录、下单主流程、关键展示验证。也就是说，能在接口层验证的尽量不要都堆到 UI 层，UI 主要验证“页面是否真的可操作、关键流程是否真的能走通”，接口则承担大部分业务规则校验。

### 7. APP 自动化和 Web UI 自动化有什么不同
- tags: APP自动化, UI自动化, 高频
- difficulty: medium
- prompt: APP 自动化和 Web UI 自动化相比，有哪些主要区别？
- answer: APP 自动化和 Web UI 自动化底层思路类似，都是元素定位、操作和断言，但 APP 场景会更复杂。比如设备碎片化更多，要考虑不同机型、系统版本、分辨率；还会涉及安装、卸载、权限弹窗、前后台切换、手势操作、弱网、中断场景等。Web 更偏浏览器兼容和页面加载，APP 则更偏设备环境和系统交互，所以 APP 自动化通常在环境控制和稳定性治理上更难一些。

### 8. APP 自动化常见难点有哪些
- tags: APP自动化, 稳定性, 高频
- difficulty: medium
- prompt: 做 APP 自动化时，常见难点有哪些？
- answer: 常见难点主要有几个：第一，元素定位稳定性差，尤其是复杂组件或混合页面；第二，权限弹窗、升级弹窗、系统弹窗等干扰多；第三，设备和系统版本碎片化明显；第四，执行环境搭建和维护成本高；第五，弱网、来电、切后台这类中断场景更复杂。整体上 APP 自动化的难点不是单点技术，而是环境、设备、系统交互一起叠加。

### 9. 接口自动化框架你会怎么设计
- tags: 接口自动化, 框架设计, 中高频
- difficulty: medium
- prompt: 如果让你设计一个接口自动化测试框架，你会怎么拆？
- answer: 我一般会拆成几个层次：用例层、请求封装层、数据管理层、断言层、配置层、日志报告层。再往工程化一点，会加环境切换、鉴权处理、公共前后置、数据库校验、Mock 支持、CI 集成等能力。我的思路是让用例尽量只关心业务输入和业务预期，把请求发送、鉴权、日志、重试这些公共能力沉到框架里。

### 10. APP 自动化里为什么很多团队只保留少量核心用例
- tags: APP自动化, 自动化策略, 中高频
- difficulty: medium
- prompt: 为什么很多团队不会给 APP 做特别大规模的自动化，而是只保留少量核心用例？
- answer: 因为 APP 自动化维护成本通常比较高，稳定性也容易受设备、系统、网络、弹窗等因素影响。如果把大量普通场景都压到 APP 自动化，投入和收益往往不成正比。所以很多团队会保留少量核心冒烟和主流程用例，把更多业务规则下沉到接口层或单元层验证，这样整体自动化 ROI 会更合理。
